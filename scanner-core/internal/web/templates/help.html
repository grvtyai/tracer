Operator Guide

This page is the built-in working guide for tracer and startrace. It explains what run states mean, how to troubleshoot failed plugins and how reevaluation should be used in day-to-day operation.

Run Status

Completed
All planned work finished cleanly or warnings were explicitly accepted by the operator.
Needs attention
One or more jobs failed or the run produced reevaluation hints. Review the failed jobs, fix what matters, then rerun or accept warnings when the remaining gaps are acceptable.
Running
The scan is still active. The run becomes clickable after completion.
Failed
The run could not finish. This should stay rare and normally means a bigger setup or execution issue.

When a Run Needs Attention

  • Open the run and read the yellow warning box first.
  • Check the failed job list to see which plugin failed, on which host and with which error text.
  • If the failed plugin matters for your goal, fix the tool, path or target issue and run the scan again.
  • If the missing plugin result is acceptable and the rest of the data is good enough, use `Accept warnings and complete`.
  • Use reevaluation when uncertainty should be checked later rather than blocking the current analysis.

Plugin Troubleshooting

  • naabu: check binary availability, correct active interface and reachable scope.
  • nmap: check privilege level, host reachability and service ports discovered earlier in the run.
  • scamper: traceroute-style reachability can fail even when later TCP probes still work. This is common in private networks and not always critical.
  • httpx and zgrab2: verify the host really offers HTTP on the scanned port and that redirects or TLS behavior are acceptable.
  • zeek: passive ingest depends on visible traffic and a valid log directory. Zero passive findings can still be fine.

Reevaluation

  • `Reevaluate all Hosts` opens a new scan configuration seeded with the current run scope.
  • `Timebased Reevaluation` stores a future reevaluation request for the same run scope. The schedule is persisted already; automatic execution is the next planned step.
  • `Reevaluate this Host` on an asset opens a new scan configuration focused on just that host.
  • The asset-level reevaluate checkbox keeps a host marked for future follow-up even when run-wide reevaluation is off.

Asset Overrides

  • Observed data always stays intact.
  • `Edit Host` lets you add manual labels such as device type, connection type, tags and notes.
  • Use overrides when you know more than the scanner can safely infer, for example that a device is definitely Wi-Fi even though the scan can only guess.

Analytics

  • Analytics groups runs and assets into a dashboard-friendly shape.
  • Device type and connection type become much more useful once manual overrides are maintained.
  • For private networks this is the basis for later views such as smartphones, workstations, servers and IoT groups.
